# Tracking

Effectiveness tracking for design-please engagements. Each run of a workflow (or a major skill sequence) gets a log entry so the methodology can evolve based on what actually worked.

## Files

- `effectiveness-log.md` — per-engagement log entries. Append-only; most recent on top.

## When to log

- After a full workflow completes (`new-engagement`, `design-foundations`)
- After a significant standalone skill run (e.g., a design-encode run that required multiple iterations)
- When an engagement outcome surfaces a pattern worth capturing (good or bad)

## What to log

Use the template in `effectiveness-log.md`. Capture:
- Project type and scope
- Skills used
- Dimensions targeted and before/after scores
- Facilitation level (self-serve vs. facilitated)
- Artifacts produced
- Key insights — what worked, what was difficult, would-recommend rating

## How the log feeds improvement

Effectiveness entries are raw input for methodology updates. When a pattern appears across multiple entries (e.g., "users consistently underestimate design system scope" or "the feel-direction interview needs more cultural-reference prompting"), it signals a potential SYSTEM.md or SKILL.md edit.

Do not edit SYSTEM.md or SKILL.md based on a single entry. Look for repeated signal across three or more engagements before changing methodology.
